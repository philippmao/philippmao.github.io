In a pentest I came across a java-rmi service. Unfortunately during the engagement I was unable to exploit the service. (exposed on the domain controller..) So I decided to download the trial version and install the software locally to see if I could find a suitable gadget. 
  
Spoilers: I did not find anything, however the process may be of interest to other people looking to exploit Java deserialization.

Checking the installation directory there is a Server.jar file. Decompiling the file in jd-gui quickly shows that the interfaces for the Java-RMI service are implemented in there. 

There is also a main class present, which starts the Java-RMI listener. 
![Task Manager](/images/b1/main_server.png)

In order to run the RMI service and get debugging output, I kill the program and start the server manually with Java using the following command:

```cmd
C:\Program Files (x86)>java.exe -Xdebug -Xrunjdwp:transport=dt_socket,server=y,suspend=n,address=5005 -Djava.security.manager=default -Djava.security.policy=="C:\Program Files (x86)\server.policy" -Djava.library.path=.\lib -Djava.rmi.server.logCalls=true  -cp ".\Server.jar;.\lib\*;client\plugins\*" com.server.core.Server
```

Most of the command line options are obtained by trying to start the server and getting an error message. (like this class is not present in the classpath).

Afterwards we have the RMI server up and running and I can see the standard output.

# Insecure Deserialization

One of the dangers of an exposed Java-RMI service is the fact that exposed functions which have non-primitive arguments (like an object), will need to be deserialized. The object which is deserialized is under the control of the attacker. 

To exploit this first I will need to find a function using non-primitive parameters. There are plenty present, I will use the function:
```java
boolean isInsatlled(java.rmi.server.ObjID paramObjID)
```

This function is exposed on the Serverdatabasemanager registry:
![Task Manager](/images/b1/target_function.png)

To confirm the fact that this is vulnerable I will use rmiscout (https://github.com/BishopFox/rmiscout) with a universal gadget which triggers a DNS callback.

```bash
./rmiscout.sh exploit -s 'boolean isInsatlled(java.rmi.server.ObjID paramObjID)' -p ysoserial.payloads.URLDNS -c "http://qc6495eq1ezax1pijwa6mj476ycp0e.burpcollaborator.net" -n Serverdatabasemanager 192.168.100.202 5120
```

Checking the burp collaborator, a DNS callback is observed:
![Task Manager](/images/b1/collaborator.png)

# RCE Gadgets

With the DNS callback it is clear that the server deserializes arbitrary objects. Now the goal is to obtain code execution. To do this there need to be gadgets which can be used to trigger RCE. RMIScout implements a few ysoserial gadget chains. Unfortunately trying these it is quickly appearant that none of these gadgets are present in the classpath.

Trying the commonscollections5 gadget:
```bash
./rmiscout.sh exploit -s 'boolean isInsatlled(java.rmi.server.ObjID paramObjID)' -p ysoserial.payloads.CommonsCollections5 -c "ping 192.168.2.68" -n Serverdatabasemanager 192.168.100.202 5120
```

The resulting stack trace indicates that the class to be deserialized was not found in the classpath. 
![Task Manager](/images/b1/notfound.png)

The classpath of the application contains quite a few Jar files. There may be some gadget chain present.

To try and find these gadgets I use the tool gadgetinspector (https://github.com/JackOfMostTrades/gadgetinspector).
After running the tool it finds three possible gadget chains. I will ignore the last gadget chain since the sink is FileInputStream, which reads a file on the filesystem. The two other gadgets end in a invoke call, which could be used for RCE.
![Task Manager](/images/b1/inspector.png)

Checking the source code of org.apache.log4j.pattern.LogEvent and it's implementation of the readObject function. (This function is called when such a object is deserialized)
![Task Manager](/images/b1/log4j_1.png)

As can be seen, right away the readLevel function is called on the input stream.

The relevant source code of the reaLevel function:

![Task Manager](/images/b1/log4j_2.png)

As can be seen on line 413, the classname is retrieved from the input stream and in line 419 used to retrieve the class. This means wathever is placed in the inputstream is not actually what gets invoked. The method is retrieved using the classloader's class. Unfortunately we will not be able to execute our own code with this gadgetchain. The other gadgetchain is almost identical.

So this is it then... Well not quite. There are still a few possible attack vectors. There may be gadgets present, which gadgetinspector did not find. Additionally there may be vulnerabilities in the handling of the RMI procedure calls. However I felt like I reached the point of dimishing returns and decided to call it here.
